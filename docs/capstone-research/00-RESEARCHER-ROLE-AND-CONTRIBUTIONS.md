# Researcher Role in a 4-Member Capstone Team

## Typical team split (example)

| Member | Common role | Main outputs |
|--------|-------------|--------------|
| Member 1 | Project Lead / Backend | API, database, server |
| Member 2 | Frontend Developer | React UI, integration |
| Member 3 | **Researcher** | Chapters 1–3, literature, requirements, analysis |
| Member 4 | QA / Documentation / Design | Testing, manuals, diagrams, deployment |

Roles overlap; everyone should understand the system. The **researcher** owns the *academic and requirements backbone* that justifies *why* the system exists and *how* it was built.

---

## What YOU contribute as Researcher

### 1. Academic foundation (≈40% of your load)
- Write **Chapters 1–3** (Introduction, Review of Related Literature, Methodology)
- Maintain consistent **APA/IEEE citations** and reference list
- Align problem statement and objectives with the **actual system** your team built (EvaTrack / EvacConnect)

### 2. Evidence and justification (≈25%)
- Find and summarize **related studies** (disaster management, evacuation tracking, QR, geotagging, emergency systems)
- Connect each feature in your app to a **gap in literature** or **local practice**

### 3. User-centered requirements (≈20%)
- Plan and run **interviews/surveys** with barangay officials, DRRM, center staff
- Document **pain points** and **functional requirements**
- Share findings with developers so features match real needs

### 4. System analysis documentation (≈15%)
- Describe **current manual process** (AS-IS)
- Propose **automated process** (TO-BE) with flowcharts
- Support the group’s **SRS / scope** and **testing scenarios**

## Deliverables checklist (your name on these)

- [ ] Chapter 1: Introduction (problem, objectives, scope, significance)
- [ ] Chapter 2: Related Literature and Studies (with reference list)
- [ ] Chapter 3: Methodology (SDLC, tools, data gathering, evaluation)
- [ ] Interview/survey instruments + raw summaries (appendix)
- [ ] Requirements findings document
- [ ] AS-IS / TO-BE process flows
- [ ] Help teammates write **Significance** and **Abstract** from your chapters
- [ ] Coordinate with QA: test cases should trace back to **objectives** and **requirements**

## How to work with your 3 teammates

| Teammate | You ask them for… | You give them… |
|----------|-------------------|----------------|
| Backend | Feature list, API list, database ERD | Requirements, use cases |
| Frontend | Screen list, user roles | User stories, flow diagrams |
| QA / Docs | Test results, user manual draft | Objectives ↔ test traceability |
| Leader | Timeline, final chapter format (school template) | Weekly research progress, citations |

---

## Tips for capstone defense

1. **Do not copy-paste** literature summaries without citation — use your own words after reading.
2. Every **objective** in Chapter 1 should appear in **Methodology** (how you achieved it) and **Results** (evidence).
3. Bring **interview quotes** or survey charts to defense — judges trust primary data.
4. Your `.docx` template from school: paste content from the markdown files in `docs/capstone-research/` and apply heading styles there.
